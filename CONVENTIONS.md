# Spec-Driven Development Conventions

> **If you are an AI coding agent, you MUST FOLLOW THESE RULES. This is the PROJECT-SPECIFIC convention, so NO NEGOTIATION!**

## Core Principles

- The spec is the source of truth for a feature, not the code
- Specs describe INTENT (why it exists, what problem it solves) and BOUNDARIES (what files it touches), not implementation details
- Code describes HOW. Specs describe WHY, WHAT, and WHERE.
- Every feature MUST have a spec folder before implementation begins
- The original prompt or requirement MUST be preserved. It is the source of truth.
- Each dev uses their own agents and workflows, but ALL must follow the same spec convention
- When modifying a feature, the spec MUST be updated (plan, work-logs, or README as needed)

---

## Spec Structure

### Location

All specs live in a `.specs/` folder at the project root. Each feature gets its own subfolder inside `.specs/`.

### Folder Naming Pattern

```
{index}--{kebab-case-title}--{YYYY-MM-DD}
```

- `index`: 3-digit zero-padded number (001, 002, ..., 013)
- `title`: kebab-case feature name
- `YYYY-MM-DD`: creation date, fixed at the time the spec was created
- Separators: double dashes `--`

Examples:

- `001--user-authentication--2026-03-01`
- `002--billing-integration--2026-03-05`
- `013--dark-mode-toggle--2026-03-15`

### Folder Structure

```
project-root/
├── .specs/
│   ├── 001--user-authentication--2026-03-01/
│   │   ├── README.md          📋 Status, owner, original prompt
│   │   ├── plan.md            🗺️  Agent-generated implementation plan
│   │   └── work-logs.md       📝 Timeline & change history (optional)
│   │
│   ├── 002--billing-integration--2026-03-05/
│   │   ├── README.md
│   │   ├── plan.md
│   │   └── work-logs.md
│   │
│   └── 003--notification-system--2026-03-10/
│       ├── README.md
│       └── plan.md            (no work-logs — single-session feature)
│
├── src/                        ← your codebase (untouched by this convention)
└── ...
```

## Files Inside Each Feature Folder

### README.md (required)

This is the feature's identity card. Short, scannable, no fluff.

MUST contain:

- **Status**: one of `draft`, `in-progress`, `completed`, `on-hold`, `deprecated`
- **Owner**: who is responsible for this feature
- **Created / Updated**: dates
- **Original prompt/requirement**: the exact prompt or requirement that initiated this feature. This is the most important field in the entire spec. It preserves original intent. MUST NOT be edited or paraphrased.
- **Summary**: 2-3 sentences max describing what this feature does and why it exists

Template:

```markdown
# Feature: [Name]

**Status**: draft | in-progress | completed | on-hold | deprecated
**Owner**: @developer
**Created**: YYYY-MM-DD
**Updated**: YYYY-MM-DD

## Original Prompt

> [Paste the exact original prompt or requirement here. Do not edit or paraphrase.]

## Summary

[2-3 sentences. What this feature does and why it exists.]
```

### plan.md (required)

Generated by the dev's preferred agent, framework, or workflow (Cursor, Claude Code, BMAD, or whatever the team uses). The convention does NOT dictate format. Each dev uses whatever planning tool they prefer.

The plan MUST meet these constraints:

- MUST be under 500 lines
- MUST include a section listing all files and folders this feature touches, so other agents know the scope
- SHOULD include the implementation approach, architecture decisions, and task breakdown
- Specific headings, structure, and format are up to the dev or agent that generates it

### work-logs.md (optional)

A timeline of work done on this feature. Useful when:

- The feature spans multiple days or sessions
- Multiple devs contribute to the same feature
- You want to track what was tried, what failed, what changed

Append-only format:

```markdown
# Work Logs

## YYYY-MM-DD — @developer
- What was done
- Decisions made
- Blockers encountered

## YYYY-MM-DD — @other-developer
- Continued from previous session
- Updated plan to reflect new approach
```

## Rules

### For Developers

1. **New features**: create the spec folder BEFORE implementation. Write the README with the original prompt. Generate the plan using your preferred agent or workflow.
2. **Continuing work**: read the spec folder (README + plan) BEFORE resuming. Add entries to work-logs if applicable.
3. **Modifying someone else's feature**: read the spec folder first. Update the plan if the modification changes scope or architecture. Add yourself to work-logs.
4. **Completing a feature**: update README status to `completed`. Update the `Updated` date.

### For AI Agents

Before working on ANY feature:

```
1. READ .specs/{feature-folder}/README.md — understand the original intent
2. READ .specs/{feature-folder}/plan.md — understand the scope and related files
3. Only touch files listed in the plan's related files/folders section
4. If changes require files NOT listed in the plan → flag to human, propose plan update

After working:
5. UPDATE work-logs.md with what was done (append, don't overwrite)
6. UPDATE plan.md if the implementation approach changed
7. UPDATE README.md dates if applicable
8. NEVER modify the "Original Prompt" section in README.md
```

The Original Prompt section is sacred. It represents the human's original intent. Agents MUST NOT modify, paraphrase, or "improve" it.

---

## Cross-Feature Concerns

- When a feature touches files that belong to another feature's plan, BOTH specs should note the overlap
- If your project has many features, consider adding a top-level `.specs/README.md` as a feature index

---

## Granularity Guidelines

- One spec per feature or initiative, not per file and not per function
- The test: "Could I explain this to a new dev in under 2 minutes from the README?" If yes, granularity is right.
- If a feature grows too large, split it into sub-features with their own spec folders

---

## Anti-Patterns

**Skipping the original prompt.** The whole point is preserving intent. If you lose the prompt, you lose the "why."

**Over-engineering the plan.** The plan should be useful, not exhaustive. Under 500 lines. If you need more, the feature is too big. Split it.

**Not updating specs when modifying features.** If you change the code but not the spec, the next dev or agent works from stale information. This is the number one failure mode.

**Treating specs as documentation.** Specs are not docs you write after the fact. They are the starting point, written BEFORE code and updated DURING development.

**Editing the original prompt.** The prompt is a historical record. If requirements change, add a new section or note in work-logs. Don't rewrite history.
